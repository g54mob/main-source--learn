# Windsurf Workflows — Usage Guide

## How to Trigger

In the Cascade chat panel, type `/` followed by the workflow name:

```
/init
/build-phase
/audit-phase
/decouple-check
/cross-phase-mod
```

A dropdown appears as you type. Select the workflow → Cascade reads the `.md` file and executes steps in order.

---

## Workflow Reference

| Command | When to Use | What It Does |
|---------|------------|--------------|
| `/init` | Once per new project | Reads ALL `.cs` in Assembly-CSharp + `.stub` file, generates ARCHITECTURE.md, GOAL.md, PhaseMap.md, StructureMap.md, Estimate.md, SystemPortabilityMap.md, surfer.md |
| `/build-phase` | Every new phase | Reads all reference docs + original source, generates all scripts + GUIDE.md + FLOW.md + Dependency.md per system, then auto-runs audit/decouple/cross-phase checks |
| `/audit-phase` | After any delivery | Method-by-method comparison against original source + architecture rule violations scan + doc consistency check |
| `/decouple-check` | After modifying any system | Scans a `_-Systems/` folder for concrete cross-system imports, classifies as FREE/interface/concrete, suggests fixes |
| `/cross-phase-mod` | Before typing a new phase | Checks if the new phase references methods/fields that don't exist in earlier phases yet, generates exact code diffs |

---

## Typical Flow

```
New project:
  /init  →  generates all docs from raw source

Per phase:
  /build-phase      →  generates everything for one phase
    ├── /audit-phase       (auto-runs as step 16)
    ├── /decouple-check    (auto-runs as step 17)
    └── /cross-phase-mod   (auto-runs as step 18)

Ad-hoc:
  /decouple-check   →  after editing any system
  /audit-phase      →  after any manual changes
  /cross-phase-mod  →  before hand-typing a new phase
```

## Key Facts

- **Only files in `.windsurf/workflows/` become slash commands** — anything else in `.windsurf/` is just regular files
- **Workflows persist across conversations** — they're `.md` files on disk, not chat memory
- **Edit anytime** — changes to workflow `.md` files take effect on next invocation
- **`// turbo` annotation** — add above a step to auto-run safe terminal commands without approval
- **Cascade asks clarifying questions** — workflows with `{PROJECT}` or `{PHASE}` placeholders prompt you for values
- **Each workflow is standalone** — `/decouple-check` works independently, doesn't require `/build-phase` first
- **`/build-phase` chains the others** — steps 16-18 run audit, decouple, and cross-phase checks automatically

<!-- ==== FILE: .windsurf/workflows/audit-phase.md ==== -->
---
description: Post-delivery self-audit — method-by-method source comparison + architecture rule check
---

## Setup

1. Ask: "Which project?" → `{PROJECT}`. Ask: "Which phase?" → `{PHASE}`

## Source Fidelity

2. For every original source file in `MAIN-SOURCE/{PROJECT}/` mapped to this phase:
   - Read original line-by-line
   - Compare every public method, field, interface against `LEARN/{PROJECT}/phase-{PHASE}/`
   - Log gaps: `| Original Method | Status | Notes |`
3. Fix all gaps before proceeding

## Architecture Rules

4. Scan every `.cs` in `LEARN/{PROJECT}/phase-{PHASE}/` for:
   - `FindObjectOfType` → replace with [SerializeField] / Owner chain / GameEvents
   - Public methods nobody external calls → demote to private/protected
   - Missing `// purpose:` on Raise/Subscribe
   - SO_ with methods → move to consumer or SO_XxxExt
   - `[SerializeField] public` → make private + Get/Set
   - `{ get; set; }` → Get/Set methods (exceptions: interfaces, [Serializable], SO_, static collections)
   - Singleton not in `1-Managers/` → move
   - Raw string tags → HasTag(TagType.x)
   - Missing `[AddComponentMenu]`
   - Defensive null checks on inspector refs → remove

## Doc Consistency

5. Verify GUIDE.md filenames match actual files
6. Verify FLOW.md event registry matches GameEvents.cs
7. Verify each Dependency.md matches actual imports

## Report

8. Output: `| Check | Pass/Fail | Details |`

---
description: Build a new phase from scratch — reads all reference docs, generates scripts + documentation
---

## Setup

1. Ask: "Which project?" → `{PROJECT}`. Ask: "Which phase?" → `{PHASE}`

## Context Load

2. Read `LEARN/{PROJECT}/GOAL.md`
3. Read `LEARN/{PROJECT}/PhaseMap.md` — target phase section
4. Read `LEARN/{PROJECT}/StructureMap.md` — DataService specs for this phase
5. Read `LEARN/{PROJECT}/handTyped(latest)/` — ground truth coding style
6. Read completed phase folders in `LEARN/{PROJECT}/` — established patterns
7. Read all original source files from `MAIN-SOURCE/{PROJECT}/` for this phase (every file in PhaseMap + any unlisted)
8. Read `LEARN/{PROJECT}/phase-All/` — shared FREE infra

## Build

9. Create `LEARN/{PROJECT}/phase-{PHASE}/Scripts/0-Core/GameEvents.cs` — partial class
10. Create `LEARN/{PROJECT}/phase-{PHASE}/Scripts/2-Data/Enums/GlobalEnums{PHASE}.cs` — camelCase values
11. For each system: create `_-Systems/XxxSystem/` with all scripts + `Dependency.md` + `Test.md`
    - [AddComponentMenu], comprehensive summaries, `// →` markers, `// purpose:` on Raise/Subscribe
    - SO_ = pure data, [SerializeField] = private `_` prefix, interfaces owned by caller or implementor
12. Create `4-Utils/` + `5-Tests/` (including `Manual/*.md` with 10 pitfall rules)

## Documentation

13. Create `GUIDE.md` — beginner-friendly (What It Looks Like, folder tree, typing order, scene setup, tests, cross-phase mods with code diffs, source vs phase diff, systems & testability)
14. Create `FLOW.md` — system map, data flows (conversation prose), event registry, portability diagram
15. Update `LEARN/{PROJECT}/phase-All/7-3D/MODEL.md`, `ANIM.md`, `WORLD.md` with `## Phase {PHASE}` sections

## Verify

16. Run `/audit-phase` on this phase
17. Run `/decouple-check` on each system
18. Run `/cross-phase-mod` for this phase
19. Append entry to `LEARN/{PROJECT}/surfer.md`

<!-- ==== FILE: .windsurf/workflows/cross-phase-mod.md ==== -->
---
description: Check if new phase references methods/fields missing from earlier phases
---

## Setup

1. Ask: "Which project?" → `{PROJECT}`. Ask: "Which phase?" → `{PHASE}`

## Scan

2. Read all `.cs` in `LEARN/{PROJECT}/phase-{PHASE}/`
3. Extract all calls/field accesses on types from EARLIER phases
4. For each: read the actual earlier phase file in `LEARN/{PROJECT}/phase-{earlier}/`
5. Flag any method/field/enum that doesn't exist yet

## Generate

6. Output: `| Earlier Phase File | What to Add | Why | Breaking? |`
7. For each: exact code diff with `// ← ADD` markers

## Update

8. Add to `LEARN/{PROJECT}/phase-{PHASE}/GUIDE.md` → "Modifications to Earlier Phases" section

<!-- ==== FILE: .windsurf/workflows/decouple-check.md ==== -->
---
description: Scan a system for concrete cross-system deps — the #1 rework cause
---

## Setup

1. Ask: "Which project?" → `{PROJECT}`. Ask: "Which system folder?" (or "all in phase X")

## Scan

2. For every `.cs` in the target `_-Systems/XxxSystem/`:
   - List `using` statements
   - Grep `Singleton<X>.Ins` (skip FREE: UIManager, DataManager)
   - Grep `GetComponent<ConcreteClass>()` (should be `<IInterface>()`)
   - Grep direct class refs from other `_-Systems/` folders
   - Grep `FindObjectOfType`
   - Check GameEvents signatures for concrete types

## Classify

3. Per dependency:
   - **FREE** (GameEvents, Singleton<T>, UIManager, DataManager, Utils.*, GlobalEnumsX, phase-All/) → OK
   - **Interface dep** (only imports an interface .cs) → OK, portable
   - **Concrete dep** (imports class from another system) → ❌ MUST FIX

## Fix

4. Per concrete dep, suggest:
   - New interface (caller-defines or implementor-defines)
   - GameEvent (fire-and-forget)
   - Bridge script (1-3 method runtime context, lives on game-specific side)
   - Static accessor (display-only queries)

## Report

5. `# SystemName — L{n} ✅/❌` with deps table

<!-- ==== FILE: .windsurf/workflows/init.md ==== -->
---
description: Analyse a raw source project and bootstrap all architecture docs from scratch
---

## Setup

1. Ask: "Which project?" → sets `{PROJECT}` (e.g., `project-0`)
2. Verify `MAIN-SOURCE/{PROJECT}/` exists and contains source scripts
3. Read `MAIN-SOURCE/entire-{PROJECT}.stub` — this is the COMPLETE file hierarchy of the original project including assets excluded due to size. Use it to understand the full project scope (3D models, audio, prefabs, scenes, etc.) even if those files aren't physically present.
4. Create `LEARN/{PROJECT}/` folder structure if it doesn't exist

## Source Analysis

5. Cross-reference the `.stub` hierarchy against `MAIN-SOURCE/{PROJECT}/` — identify:
   - Which folders are asset-only (listed in .stub but not present — 3D, audio, textures, scenes)
   - Which contains engine/third-party DLLs (skip these)
6. Read **every single `.cs` file** in `MAIN-SOURCE/{PROJECT}/Scripts/Assembly-CSharp/` (or equivalent game code folder). This is the actual game source — read ALL of them, no skipping.
   - Start with the 10 largest files (god-objects that reveal the dependency graph)
   - Then read every remaining `.cs` file
7. For each file read, extract: class name, base class, interfaces implemented, singletons, [SerializeField] fields, public methods, FindObjectOfType calls, event subscriptions/raises, direct references to other classes
8. Map the full dependency graph: who imports who, who calls who, who subscribes to whose events. Count total files.

## Generate ARCHITECTURE.md

9. Create `LEARN/{PROJECT}/ARCHITECTURE.md` with:
   - **Source Overview** — total file count, folder structure, engine version
   - **Dependency Graph** — which scripts depend on which (grouped by domain)
   - **God Objects** — files >200 lines doing 3+ things (split candidates)
   - **Singleton Registry** — every singleton, what it owns, who reads it
   - **Coupling Hotspots** — FindObjectOfType calls, direct cross-domain references, static instance access
   - **Event System** — existing event patterns (UnityEvent, C# Action, custom)
   - **Data Flow** — how data moves: SO → runtime wrapper → manager → UI
   - **Inheritance Chains** — class hierarchies worth preserving vs flattening
   - **External Packages** — third-party assets/plugins identified from DLLs and namespaces
   - **Asset Inventory** (from .stub) — 3D models, audio clips, prefabs, scenes, materials, shaders listed in the stub but not physically present. Gives complete picture of what the project uses beyond code.

## Generate PhaseMap.md

10. Group all source scripts into logical phases by dependency order:
   - Phase = set of scripts that can be built and tested independently
   - Rule: each phase depends only on earlier phases, never forward
   - For each phase: file list, "What It Looks Like When Running", difficulty rating, weight %
11. Create `LEARN/{PROJECT}/PhaseMap.md` with all phases + cross-phase modification tables

## Generate StructureMap.md

12. For each phase, spec the internal structure:
    - Which scripts become `_-Systems/` folders (3+ tightly coupled files)
    - DataService candidates (pure C# logic extractable from MonoBehaviours)
    - Interface candidates (cross-system communication points)
    - SO_ / Field_ / W prefix assignments
13. Create `LEARN/{PROJECT}/StructureMap.md`

## Generate GOAL.md

14. Copy the proven architecture rules from the general template (folder structure, naming, class responsibilities, decoupling patterns, lifecycle, C# features, testing strategy)
15. Customize with project-specific details:
    - Project name replaces `[PROJECT_NAME]`
    - Actual phase names/counts
    - Project-specific SPACE_UTIL or utility extensions (if any in source)
    - Project-specific enums, tags, layers found in source
16. Create `LEARN/{PROJECT}/GOAL.md`

## Generate Supporting Docs

17. Create `LEARN/{PROJECT}/Estimate.md` — hours per phase based on file count × complexity tier
18. Create `LEARN/{PROJECT}/SystemPortabilityMap.md` — initial L0/L1+ classification per system
19. Create `LEARN/{PROJECT}/surfer.md` — first entry: "Prompt 1: Initial source analysis"

## Verify

20. Output summary: total scripts found (code), total assets found (from .stub), phases planned, systems identified, estimated hours
