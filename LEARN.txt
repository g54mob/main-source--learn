## Event System

### Patterns Used
1. **C# `Action` / `Action<T>`** ‚Äî Primary pattern (324 matches across 120 files)
2. **`event Action<T>`** ‚Äî Standard C# event accessors on entity lifecycle (`g.cs`: pcl/pcm/pcn/pco/pcp events)
3. **No UnityEvent usage detected** in game code
4. **No custom event bus** ‚Äî direct subscription model

### Key Event Chains
- `g` entity lifecycle ‚Üí `Action<g>` events (created, started, enabled, disabled, destroyed)
- `VoxelMesh` ‚Üí `Action<hz<int3, Voxel>>` voxel change events
- `VoxelMesh` ‚Üí `Action<el>` chunk update events
- `bcx` entity ‚Üí `Action sql` parameter change event
- Limb system ‚Üí separation/destruction callbacks

# GOAL ‚Äî npcfr

## Target Architecture Rules

### 1. Folder Structure

```
npcfr-reimpl/
‚îú‚îÄ‚îÄ _-Systems/           ‚Üê folders containing 3+ tightly coupled scripts
‚îÇ   ‚îú‚îÄ‚îÄ VoxelMeshCore/
‚îÇ   ‚îú‚îÄ‚îÄ EntityCore/
‚îÇ   ‚îú‚îÄ‚îÄ LimbSystem/
‚îÇ   ‚îú‚îÄ‚îÄ etc.
‚îú‚îÄ‚îÄ DataServices/        ‚Üê pure C# classes (no MonoBehaviour) with extractable logic
‚îú‚îÄ‚îÄ Interfaces/          ‚Üê cross-system communication contracts
‚îú‚îÄ‚îÄ SO_/                 ‚Üê ScriptableObject assets
‚îú‚îÄ‚îÄ Field_/              ‚Üê serializable data structs
‚îú‚îÄ‚îÄ W_/                  ‚Üê runtime wrappers around SO_ data
‚îú‚îÄ‚îÄ Enums/
‚îú‚îÄ‚îÄ Tests/
‚îî‚îÄ‚îÄ [Phase folders as needed]
```

### 2. Naming Conventions

| Element | Convention | Example |
|---------|-----------|---------|
| ScriptableObject | `SO_` prefix | `SO_IconData` |
| Data struct | `Field_` prefix | `Field_Voxel`, `Field_SpringData` |
| Runtime wrapper | `W_` prefix | `W_ToolbarSlot` |
| Interface | `I_` prefix | `I_Creature`, `I_Limb` |
| System folder | `_-Systems/[Name]/` | `_-Systems/VoxelMeshCore/` |
| Service (pure C#) | `[Name]DataService` | `LimbDamageDataService` |
| MonoBehaviour | PascalCase, no prefix | `VoxelMesh`, `AbstractLimb` |
| Enum | PascalCase | `LVAEffectorFeedbackType`, `TimerTimeType` |

### 3. Class Responsibilities

- **MonoBehaviour**: Only Unity lifecycle + serialized references. Delegates logic to DataServices.
- **DataService**: Pure C# class. Testable without Unity. Receives data, returns results.
- **ScriptableObject**: Immutable configuration. Never mutated at runtime.
- **Wrapper (W_)**: Mutable runtime copy of SO_ data. Owned by a single manager.
- **Interface**: Defines communication contract between systems. Never concrete class references across system boundaries.

### 4. Decoupling Patterns

#### Cross-System Communication
- Systems communicate **only** through interfaces
- No direct class references across `_-Systems/` folder boundaries
- Zenject binds interfaces to implementations in Installers

#### Event Pattern
- Use `Action` / `Action<T>` for events (matches source style)
- Event owners declare events; subscribers inject the owning interface
- No global event bus ‚Äî direct injection only

#### Dependency Direction
- Each phase depends only on earlier phases (see PhaseMap)
- No circular dependencies between `_-Systems/` folders
- Utility code (P1) has zero dependencies

### 5. Lifecycle Management

- **Entity lifecycle** preserved: `bgb ‚Üí g ‚Üí bcx ‚Üí bcz ‚Üí concrete`
- Virtual hooks: `cxh/cxi/cxj/cxk/cxl` (Init/Start/Enable/Disable/Destroy)
- Behavior system: `bcw.gsu()` (Init), `bcw.gqo()` (Tick), `bcw.hls()` (Cleanup)
- Zenject `[Inject]` for constructor injection on MonoBehaviours

### 6. C# Features

- **Burst-compatible structs** for VoxelMesh (Voxels, Voxel) ‚Äî must remain blittable
- **NativeArray/NativeHashSet/NativeList** for voxel operations ‚Äî must manage allocation/disposal
- **IEquatable<T>** on value types used in NativeContainers
- **IteratorStateMachine (coroutines)** for async mesh operations
- **Extension methods** in dedicated static classes (fp.cs pattern)
- **Generics** on entity framework (bcz<TParam, TBehavior>)

## Coupling Hotspots

### 1. `cam.cs` ‚Äî Universal Dispatcher via SendMessage + Name-based GetComponent
```csharp
gameObject.GetComponent(gameObject.name).SendMessage("use");
```
This looks up a component **by the GameObject's name** and calls `use()` via SendMessage. Every interactable object must have its script name match its GameObject name. Extremely fragile.

### 2. Direct cross-domain references everywhere
- `lift.cs` directly reads `pl.GetComponent<pl>().cam.GetComponent<tryaska>()` ‚Äî three levels deep
- `hide.cs` reads `pl.GetComponent<pl>().cam`, `.camm`, `.MX`, `.MY`, `.MZ`
- `wfscare1.cs` writes to `pl.GetComponent<pl>().MX` and `pl.GetComponent<pl>().MY`
- `trigscare.cs` calls `v.GetComponent<vkl>().use()` and `c.GetComponent<cam>().dist`

### 3. Uncached GetComponent calls in Update
Almost every script calls `GetComponent<>()` inside `Update()` or `FixedUpdate()` rather than caching in `Start()`. Examples:
- `dver.cs`: `base.gameObject.GetComponent<AudioSource>()` ‚Äî called 8+ times per frame
- `cam.cs`: `GetComponent<vkl>()`, `GetComponent<dver>()` per raycast

### 4. Tag-based string comparisons
- `"Player"` tag checked in: cam, addforcescr, trigscare, wfscare1, wfcolis, wfsatr, opttrig, wfkill
- `"slomokno"` tag in cam
- `"wflook"` tag in wfscare2

### 5. `.active` property (deprecated)
Multiple scripts use `.active` instead of `.activeSelf`:
- `fl.active` in wfcode, wfzap
- `aud.active` in wfcode1
- `p.active` in wfcolis

### 2. Naming Conventions

| Element | Convention | Example |
|---|---|---|
| MonoBehaviour (system) | PascalCase, descriptive | `PlayerController`, `PhysicsDoor` |
| MonoBehaviour (wrapper) | `W_` prefix | `W_PlayerFootsteps` |

| Public API methods | PascalCase, descriptive verbs | `Use()`, `ToggleLight()`, `Call()` |
| Coroutines | PascalCase with descriptive name | `OpenDoorsSequence()`, `FlickerLoop()` |

### 3. Class Responsibilities

**MonoBehaviour scripts must do ONE thing:**
- `PlayerController` ‚Üí movement only (no camera, no audio)
- `PlayerLook` ‚Üí mouse look only
- `InteractionRaycaster` ‚Üí raycast + dispatch only
- `FlashlightController` ‚Üí flashlight hold/toggle only

**No script should:**
- Directly access another system's private state (no `pl.GetComponent<pl>().MX = -90f`)
- Use `SendMessage()` ‚Äî use `IInteractable.Use()` instead
- Use `GetComponent(string)` by name ‚Äî use typed interface
- Call `GetComponent<>()` in `Update()` ‚Äî cache in `Awake()`/`Start()`

### 4. Decoupling Patterns

#### Replace SendMessage with IInteractable
```csharp
// BEFORE (cam.cs)
gameObject.GetComponent(gameObject.name).SendMessage("use");

// AFTER (InteractionRaycaster.cs)
if (hit.transform.TryGetComponent<IInteractable>(out var interactable))
    interactable.Use();
```

#### Replace direct field writes with interface/events
```csharp
// BEFORE (wfscare1.cs)
pl.GetComponent<pl>().MX = -90f;
pl.GetComponent<pl>().MY = 10f;

// AFTER ‚Äî via event or interface
public static event System.Action<Vector2> OnForceLookDirection;
OnForceLookDirection?.Invoke(new Vector2(-90f, 10f));

// PlayerLook subscribes
OnForceLookDirection += (dir) => { _mx = dir.x; _my = dir.y; };
```

#### Replace state integers with enums
```csharp
// BEFORE (lift.cs)
public int x; // 0, 1, 2, ... 23

// AFTER
public enum ElevatorState { Idle, Ascending, DoorsOpening, WaitingForPlayer, ... }
private ElevatorState _state = ElevatorState.Idle;
```

#### Cache GetComponent calls
```csharp
// BEFORE (dver.cs) ‚Äî called 8+ times per frame
base.gameObject.GetComponent<AudioSource>().Play();

// AFTER
private AudioSource _audio;
private void Awake() => _audio = GetComponent<AudioSource>();
private void PlayAudio() => _audio.Play();
```

### 5. Lifecycle Rules

- `Awake()` ‚Äî cache components, resolve references
- `Start()` ‚Äî initialize state, subscribe to events
- `OnDestroy()` ‚Äî unsubscribe from events
- `Update()` ‚Äî input reading, state checks (no heavy logic)
- `FixedUpdate()` ‚Äî physics only (`Rigidbody.AddForce`)
- Coroutines ‚Äî sequenced events with `WaitForSeconds`

### 6. C# Features to Adopt

| Feature | Use Case |
|---|---|
| `enum` | All state machines (elevator, scare events, door states) |
| `[SerializeField] private` | All inspector-assigned fields (replace `public GameObject`) |
| `TryGetComponent<T>()` | Replace `GetComponent<T>() != null` checks |
| `switch` expressions | State machine transitions |
| `readonly` fields | Cached component references |
| `string.Empty` / `nameof()` | Replace magic strings |
| Properties | Replace direct field access for external reads |

### 7. Tags & Layers

**Existing tags (preserve):**
- `Player` ‚Äî player collider
- `slomokno` ‚Äî breakable door glass (special interaction path in cam.cs)
- `wflook` ‚Äî wrong floor look target (raycast check in wfscare2)

**No custom layers identified** ‚Äî only default physics LayerMasks.

### 8. Testing Strategy

| Level | What to Test | How |
|---|---|---|
| Unit | `CombinationLockLogic` ‚Äî digit increment, code validation | NUnit, no MonoBehaviour needed |
| Unit | `ElevatorStateMachine` ‚Äî state transitions | NUnit, pure C# |
| Unit | `PlayerInputService` ‚Äî input ‚Üí vector mapping | NUnit with mock input |
| Integration | `IInteractable` dispatch ‚Äî raycaster finds and calls correct target | PlayMode test with scene setup |
| Integration | Door unlock flow ‚Äî key pickup ‚Üí door enabled | PlayMode test |
| Manual | Full horror sequences ‚Äî timing, audio, camera | Playtest checklist per phase |

### 9. Performance Guidelines

- **Cache all GetComponent calls** ‚Äî current code calls GetComponent 30+ times per frame across all scripts
- **Room culling** ‚Äî preserve opt/opttrig system, it's the only optimization
- **Single scene** ‚Äî no scene splitting needed (project is small)
- **AudioSource pooling** ‚Äî not needed (low audio source count)
- **No object pooling needed** ‚Äî no runtime instantiation found

### 10. Migration Order

1. **Phase 1 first** ‚Äî `pl`, `cam`, `fl` are the foundation everything depends on
2. **Create `IInteractable` interface before Phase 2** ‚Äî all interactables need it
3. **Phase 2 interactables** adopt `IInteractable` immediately
4. **Phase 3 elevator** ‚Äî extract state machine, adopt SO config
5. **Phase 4** ‚Äî straightforward renames, minimal logic changes
6. **Phase 5** ‚Äî largest phase, but scripts are mostly independent triggers
7. **Phase 6** ‚Äî menu last, bind to `SO_GameSettings`
